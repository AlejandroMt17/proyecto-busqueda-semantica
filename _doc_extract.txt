1. Resumen Ejecutivo y Objetivo
 
El presente documento define el plan técnico completo para construir un Sistema de Búsqueda Semántica en Batch usando Apache Spark. A diferencia de la búsqueda por palabras clave, este sistema transforma cada fragmento de texto en un vector denso de 384 dimensiones y permite encontrar documentos por significado semántico.
 
Una consulta como "problemas de motor" puede encontrar documentos sobre "falla de combustión" aunque no compartan ninguna palabra. Esto se logra mediante el modelo sentence-transformers/all-MiniLM-L6-v2 aplicado de manera distribuida sobre un corpus de 500,000+ documentos.
 
Objetivo Principal
Procesar 500,000+ documentos (Wikipedia ES / arXiv) de forma distribuida con Apache Spark.
Generar embeddings densos de 384 dimensiones usando Pandas UDFs en Spark.
Indexar los vectores en Elasticsearch con campo dense_vector y similitud coseno.
Demostrar distribución real de carga entre 3 laptops (1 Master + 2 Workers).

Paso 2 — Implementar la función de extracción de texto
La función debe manejar los tres tipos de archivo: PDF (Tika), HTML (BeautifulSoup) y TXT (lectura directa). Se llamará desde la Pandas UDF de extracción.
def extract_text(file_path: str, file_type: str) -> str:
    if file_type == 'pdf':
        from tika import parser
        result = parser.from_file(file_path, 'http://192.168.1.100:9998/tika')
        return result.get('content', '') or ''
    elif file_type == 'html':
        from bs4 import BeautifulSoup
        with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
            return BeautifulSoup(f.read(), 'lxml').get_text(separator=' ')
    else:  # txt, json
        with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
            return f.read()
 
Paso 3 — Implementar la función de limpieza de texto
La limpieza remueve artefactos comunes en datos de Wikipedia y arXiv: ecuaciones LaTeX residuales, encabezados repetitivos, caracteres de control, etc.
import re, unicodedata
def clean_text(text: str) -> str:
    if not text: return ''
    text = unicodedata.normalize('NFKC', text)
    text = re.sub(r'\s+', ' ', text).strip()       # normalizar espacios
    text = re.sub(r'\[\d+\]', '', text)           # remover citas [1]
    text = re.sub(r'https?://\S+', '', text)       # remover URLs
    text = re.sub(r'[\x00-\x1F\x7F]', '', text)  # control chars
    return text
 
Paso 4 — Implementar el chunking de 256 tokens
Los modelos de sentence-transformers tienen un límite de tokens. Se segmenta cada documento en chunks de 256 tokens con overlap de 32 para preservar contexto en los bordes.
from transformers import AutoTokenizer
tokenizer = AutoTokenizer.from_pretrained('sentence-transformers/all-MiniLM-L6-v2')
def chunk_text(text: str, chunk_size=256, overlap=32) -> list:
    tokens = tokenizer.encode(text, add_special_tokens=False)
    chunks = []
    step = chunk_size - overlap
    for i in range(0, max(1, len(tokens)), step):
        chunk_tokens = tokens[i:i + chunk_size]
        chunk_text = tokenizer.decode(chunk_tokens, skip_special_tokens=True)
        if len(chunk_text.strip()) > 30:  # filtrar chunks triviales
            chunks.append(chunk_text)
        if i + chunk_size >= len(tokens): break
    return chunks
 
Paso 5 — Implementar el ETL principal distribuido con Spark
Este es el corazón del src/etl_nlp.py. Usa Pandas UDFs para que la extracción, limpieza y chunking ocurran en los workers del cluster, no en el driver.
from pyspark.sql import SparkSession
from pyspark.sql.functions import pandas_udf, explode, col, lit
from pyspark.sql.types import ArrayType, StringType, StructType, StructField
 
@pandas_udf(ArrayType(StructType([
    StructField('chunk_id', StringType()),
    StructField('raw_text', StringType())])))
def process_document(paths: pd.Series, types: pd.Series) -> pd.Series:
    results = []
    for path, ftype in zip(paths, types):
        doc_id = hashlib.md5(path.encode()).hexdigest()
        raw = extract_text(path, ftype)
        clean = clean_text(raw)
        chunks = chunk_text(clean)
        results.append([
            {'chunk_id': f'{doc_id}_{i}', 'raw_text': c}
            for i, c in enumerate(chunks)])
    return pd.Series(results)

FASE 2: Inferencia Distribuida — Guía Paso a Paso
 
Paso 1 — Entender la restricción crítica de la Pandas UDF
⚠ PUNTO CRÍTICO DE EVALUACIÓN (-30% si se viola)
CORRECTO:  Aplicar el modelo dentro de la Pandas UDF → corre en workers (distribuido).
INCORRECTO: df.toPandas() → model.encode() en el driver → NO distribuido (penalización -30%).

La Pandas UDF se ejecuta en particiones del DataFrame, cada partición en un executor distinto.
El modelo debe cargarse una sola vez por executor usando el truco del atributo de función.
Spark NO puede serializar el modelo SentenceTransformer para enviarlo vía broadcast.
Solución: carga lazy dentro de la UDF con hasattr(embed, '_model')
 
Paso 2 — Implementar la Pandas UDF de embeddings
El patrón estándar para inferencia distribuida con modelos Python externos en Spark:
from pyspark.sql.functions import pandas_udf
from pyspark.sql.types import ArrayType, FloatType
import pandas as pd
 
@pandas_udf(ArrayType(FloatType()))
def embed(texts: pd.Series) -> pd.Series:
    # Carga lazy del modelo — UNA VEZ por executor, no por batch
    if not hasattr(embed, '_model'):
        from sentence_transformers import SentenceTransformer
        embed._model = SentenceTransformer('all-MiniLM-L6-v2')
    # Encode en batch — aprovecha vectorización PyTorch
    vectors = embed._model.encode(
        texts.fillna('').tolist(),
        batch_size=32,
        show_progress_bar=False,
        normalize_embeddings=True  # para similitud coseno
    )
    return pd.Series([v.tolist() for v in vectors])

Paso 2 — Implementar la carga masiva idempotente
La clave de la idempotencia es usar es.mapping.id apuntando a chunk_id. Si el documento ya existe con ese ID, Elasticsearch lo actualiza en lugar de duplicarlo.
# src/indexer_elastic.py
spark.conf.set('es.nodes', '192.168.1.100')
spark.conf.set('es.port', '9200')
spark.conf.set('es.nodes.wan.only', 'true')
 
# Leer embeddings de Fase 2 + texto de Fase 1 (join por chunk_id)
embeddings = spark.read.parquet(f's3a://data/embeddings/run_date={run_date}/')
chunks = spark.read.csv(f's3a://data/chunks/run_date={run_date}/', header=True)
docs = chunks.join(embeddings, 'chunk_id').select(
    'chunk_id', 'doc_id', 'title', 'raw_text', 'source_uri', 'ingestion_date', 'embedding'
)
 
# Escritura idempotente: es.mapping.id garantiza upsert por chunk_id
docs.write \
    .format('org.elasticsearch.spark.sql') \
    .option('es.resource', 'documents') \
    .option('es.mapping.id', 'chunk_id') \
    .option('es.write.operation', 'upsert') \
    .mode('append') \
    .save()

Paso 4 — Script de prueba de búsqueda semántica
Para la demo final, preparar un script Python que convierta una consulta en texto a un vector y ejecute una búsqueda kNN contra Elasticsearch.
from sentence_transformers import SentenceTransformer
from elasticsearch import Elasticsearch

model = SentenceTransformer('all-MiniLM-L6-v2')
es = Elasticsearch('http://192.168.1.100:9200')

def semantic_search(query: str, k: int = 10):
    query_vector = model.encode(query, normalize_embeddings=True).tolist()
    result = es.search(index='documents', body={
        'knn': {
            'field': 'embedding',
            'query_vector': query_vector,
            'k': k,
            'num_candidates': 100
        },
        '_source': ['title', 'raw_text', 'source_uri']
    })
    return result['hits']['hits']
 
# Uso:
results = semantic_search('motor failure prediction industrial')
for r in results:
    print(r['_score'], r['_source']['title'][:80])
 
Paso 5 — Orquestador run_pipeline.sh
El script orquestador ejecuta las 3 fases en secuencia, recibiendo run_date como parámetro obligatorio.
#!/bin/bash
set -e  # detener ante cualquier error
RUN_DATE=${1:-$(date +%Y-%m-%d)}
MASTER='spark://192.168.1.100:7077'
echo '[INFO] Iniciando pipeline Búsqueda Semántica para fecha: '$RUN_DATE

echo '[INFO] === FASE 1: ETL NLP ==='
spark-submit --master $MASTER --executor-memory 4g \
  --total-executor-cores 6 src/etl_nlp.py --run-date $RUN_DATE

echo '[INFO] === FASE 2: Inferencia Embeddings ==='
spark-submit --master $MASTER --executor-memory 4g \
  --total-executor-cores 6 src/spark_vectorizer.py --run-date $RUN_DATE

echo '[INFO] === FASE 3: Indexar en Elasticsearch ==='
spark-submit --master $MASTER --executor-memory 4g \
  --jars /opt/spark/jars/elasticsearch-spark-30_2.12-8.13.0.jar \
  src/indexer_elastic.py --run-date $RUN_DATE

echo '[INFO] Pipeline completado exitosamente para '$RUN_DATE

5.3 Decisiones Técnicas a Documentar
 
El documento técnico de arquitectura debe justificar explícitamente las siguientes decisiones:
 
Modelo de embedding: all-MiniLM-L6-v2 elegido por balance velocidad/calidad. 384 dims vs 768 de modelos más grandes. Corre en CPU de laptops en ~5ms/chunk. Soporte multilenguaje limitado; para corpus ES puro considerar paraphrase-multilingual-MiniLM-L12-v2.
Estrategia de chunking: 256 tokens con overlap de 32. El overlap evita perder contexto en los bordes de chunk. Documentos muy cortos (<30 tokens) se descartan. Justificación: límite del modelo vs granularidad de búsqueda.
Configuración Elasticsearch: Índice con similitud coseno en lugar de dot product (embeddings normalizados). HNSW nativo para k-NN aproximado. Un shard por defecto para cluster pequeño.
Estrategia de reindexado: Upsert por chunk_id (no borra el índice). Permite re-ejecutar sin perder históricos de semanas anteriores. Ventaja frente a delete-and-reload: si falla a mitad, los datos anteriores siguen disponibles.
Almacenamiento compartido: MinIO (compatible S3) para que todos los workers lean desde la misma fuente. Alternativa: NFS. MinIO es más realista para producción y soporta acceso concurrente nativo.
